# 线上运维手册

本手册用于线上服务器的日常运维、故障排查与恢复。默认系统为 Ubuntu，项目路径假设为 `/home/ubuntu/idle_cultivation_server`。

## 1. 常用命令速查

```bash
cd /home/ubuntu/idle_cultivation_server

# 启停服务
bash restart.sh
bash stop.sh

# 查看服务日志
tail -f server.log

# 查看服务进程
ps -ef | grep uvicorn | grep -v grep

# 查看端口占用
sudo lsof -i:8444
```

## 2. PostgreSQL 基础检查

```bash
# 启动并查看状态
sudo systemctl start postgresql
sudo systemctl status postgresql --no-pager

# 验证可连接
sudo -u postgres psql -c "SELECT version();"
```

如果出现：

`connection to server on socket "/var/run/postgresql/.s.PGSQL.5432" failed`

通常是 PostgreSQL 未启动，先执行上面的 `systemctl start postgresql`。

## 3. 确认服务端实际连接的数据库

当前代码默认连接库为 `idle_cultivation_game`（见 `app/core/config/ServerConfig.py`）。

```bash
sudo -u postgres psql -d idle_cultivation_game -c "SELECT current_database();"
sudo -u postgres psql -d idle_cultivation_game -c "\d+ player_data"
```

注意：不要在错误的库里改结构（例如误改到 `idle_cultivation`），否则服务仍会报字段不存在。

## 4. 字段缺失处理（开发/测试阶段统一做法）

当前项目仍处于开发测试阶段，出现字段缺失（如 `last_daily_reset_at`）时，统一采用“删库重建 + 重启”，不走增量补字段。

```bash
cd /home/ubuntu/idle_cultivation_server
bash stop.sh

sudo -u postgres psql -c "DROP DATABASE IF EXISTS idle_cultivation_game;"
sudo -u postgres psql -c "CREATE DATABASE idle_cultivation_game OWNER postgres;"
sudo -u postgres psql -d idle_cultivation_game -f sql/init.sql

bash restart.sh
tail -n 120 server.log
```

## 5. 清空重建数据库（允许丢数据）

用于大版本改表后快速恢复，适合测试环境。

```bash
cd /home/ubuntu/idle_cultivation_server
bash stop.sh

sudo -u postgres psql -c "DROP DATABASE IF EXISTS idle_cultivation_game;"
sudo -u postgres psql -c "CREATE DATABASE idle_cultivation_game OWNER postgres;"
sudo -u postgres psql -d idle_cultivation_game -f sql/init.sql

bash restart.sh
tail -n 120 server.log
```

## 6. 启动失败排查顺序（推荐）

1. 先看 `server.log` 最后 120 行，定位首个异常。
2. 检查 PostgreSQL 状态与可连接性。
3. 确认服务端连接库名是否为 `idle_cultivation_game`。
4. 检查关键表结构：`player_data` 是否包含 `last_daily_reset_at`。
5. 必要时执行“字段补齐”或“清库重建”。
6. 再次 `restart.sh`，确认日志出现启动完成信息。

## 7. 修复后验收清单

- `server.log` 无启动异常栈。
- `/api/docs` 可访问。
- 登录、拉取 `game/data` 正常。
- 每日任务与每日副本在跨天后可正常重置。

## 8. 运营发邮件（管理员接口）

### 8.1 获取管理员 token

```bash
curl -X POST "http://127.0.0.1:8444/api/admin/login?username=admin&password=admin123"
```

记录返回中的 `token`。

### 8.2 单人发信

```bash
curl -X POST "http://127.0.0.1:8444/api/admin/mail/send" \
  -H "Authorization: Bearer <ADMIN_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "account_id": "目标账号UUID",
    "title": "测试邮件",
    "content": "这是运营发信测试",
    "attachments": [{"item_id": "spirit_stone", "count": 50}]
  }'
```

### 8.3 批量发信

```bash
curl -X POST "http://127.0.0.1:8444/api/admin/mail/send_batch" \
  -H "Authorization: Bearer <ADMIN_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "all_accounts": true,
    "account_ids": [],
    "title": "全服补偿",
    "content": "维护补偿",
    "attachments": [{"item_id": "spirit_stone", "count": 100}]
  }'
```

说明：
- `all_accounts=true` 时忽略 `account_ids`。
- 邮箱容量满（100）时，该账号会被跳过并计入 `skipped_capacity`。

# 数据库与表结构说明

## 1. 文档目的

这份文档用于集中说明当前服务端主数据库里的核心表、职责边界和初始化责任，避免数据库信息散落在：

- `sql/init.sql`
- ORM 模型
- 需求记录
- 零散接口文档

当前数据库按职责分为三类：

- 游戏主线表
- 测试支持依赖表
- 运维系统表

---

## 2. 当前数据库与初始化责任

默认数据库名：

- `idle_cultivation_game`

当前初始化责任分成两层：

1. `sql/init.sql`
   - 作为数据库初始化总入口，只负责控制执行顺序
   - 通过 `\ir` 依次加载 `sql/schema/*.sql`

2. 应用启动初始化
   - `app/core/db/Database.py` 负责 Tortoise 初始化与安全补表
   - `app/ops/auth/Service.py` 负责确保引导运维账号与系统状态存在
   - `app/test_support/bootstrap/TestAccountSeeder.py` 负责测试账号存在性保障

这意味着：

- `init.sql` 是首轮建库真值入口
- 真实建表 SQL 已按子系统拆到 `sql/schema/`
- 应用启动会做幂等补齐
- 后续新增表时，必须同时考虑 SQL 初始化与 ORM 真值

当前拆分约定：

- `sql/schema/00_extensions_and_functions.sql`
  - 公共扩展与触发器函数
- `sql/schema/10_accounts_and_player.sql`
  - 账号与玩家主存档
- `sql/schema/20_mail.sql`
  - 邮箱系统
- `sql/schema/30_seed_test_accounts.sql`
  - 默认测试账号播种
- `sql/schema/40_ops.sql`
  - 运维系统相关表

说明：

- 不同子系统必须分文件维护，避免单个 `init.sql` 持续膨胀。
- 重要字段必须在 SQL 中直接写注释，优先解释：
  - 真值主键
  - 状态位
  - 软删字段
  - 版本号
  - token / confirm token
  - JSON 结构化快照字段

## 3. 游戏主线表

### 3.1 `accounts`

用途：

- 玩家账号表
- 保存登录身份、密码哈希、封禁状态、token 版本等账号级信息

关键字段：

- `id`：UUID 主键
- `username`：唯一用户名
- `password_hash`：密码哈希
- `token_version`：用于踢下线和单设备登录控制
- `is_banned`：永久封禁标记

关键约束与索引：

- 主键：`id`
- 唯一：`username`
- 可选唯一：`phone`
- 索引：`server_id`

### 3.2 `player_data`

用途：

- 玩家主存档表
- 保存 JSON 游戏数据和关键运行时间戳

关键字段：

- `account_id`：主键，同时关联 `accounts.id`
- `data`：完整游戏数据 JSON
- `last_online_at`：最近在线时间
- `last_daily_reset_at`：最近每日重置时间
- `game_version`：存档版本号

关键约束与索引：

- 主键：`account_id`
- 外键：`account_id -> accounts.id`
- 索引：`updated_at`、`last_online_at`、`last_daily_reset_at`、`server_id`、`game_version`

### 3.3 `mail_data`

用途：

- 玩家邮箱数据表
- 支持软删、已读、已领奖、附件奖励发放

关键字段：

- `mail_id`：主键
- `account_id`：关联目标玩家
- `attachments`：附件列表 JSON
- `is_read`、`is_claimed`、`is_deleted`：邮件状态字段

关键索引：

- `account_id`
- `created_at`
- `is_deleted`

## 4. 测试支持说明

当前测试支持没有单独新表，主要复用：

- `accounts`
- `player_data`
- `mail_data`

测试相关逻辑主要体现在：

- `/api/test/*` 接口
- 默认测试账号播种
- pytest 会话对测试账号并发占用的保护

注意：

- 同一测试账号不允许被多个 pytest 进程并发使用
- 这是测试约束，不是数据库模型约束

## 5. 运维系统表

### 5.1 `ops_users`

用途：

- 运维后台账号表
- 保存运维用户名、密码哈希、角色、权限和启用状态

关键字段：

- `id`
- `username`
- `password_hash`
- `role`
- `permissions`
- `is_active`
- `last_login_at`

说明：

- 一期默认引导账号由启动逻辑保证存在
- 不再依赖硬编码管理员直接作为正式方案

### 5.2 `ops_system_state`

用途：

- 全局运维系统状态表
- 当前一期用于登录闸门真值

关键字段：

- `id`
- `login_gate_enabled`
- `login_gate_updated_by`
- `login_gate_updated_at`
- `login_gate_note`

说明：

- 当前设计上至少保留一条 `id=1` 的全局状态记录
- 游戏登录接口会直接读取这里的 `login_gate_enabled`

### 5.3 `ops_login_whitelist`

用途：

- 登录闸门白名单账号表

关键字段：

- `account_id`
- `account_username_snapshot`
- `note`
- `created_by`

说明：

- 真实判定按 `account_id`
- `account_username_snapshot` 只用于展示与排查
- 默认会把测试账号 `test`、`test2` 加入白名单
  - `sql/schema/40_ops.sql` 会在首轮建库时插入
  - `OpsAuthService.ensure_bootstrap_data()` 会在应用启动时幂等补齐

### 5.4 `ops_audit_logs`

用途：

- 运维审计日志表
- 记录操作者、动作类型、目标范围、请求载荷、结果和来源信息

关键字段：

- `operator_user_id`
- `operator_username`
- `action_type`
- `target_scope`
- `target_payload`
- `request_payload`
- `result`
- `reason_code`
- `ip`
- `user_agent`

说明：

- 审计日志默认长期保留
- 需要依赖分页、索引和筛选字段

### 5.5 `ops_auth_tokens`

用途：

- 运维 token 会话跟踪与撤销

关键字段：

- `user_id`
- `token_jti`
- `expires_at`
- `revoked_at`

说明：

- 与玩家 token 体系隔离
- 用于支持登出与主动失效

### 5.6 `ops_action_confirms`

用途：

- 高危 preview/confirm 两段式动作确认表

关键字段：

- `operator_user_id`
- `action_type`
- `confirm_token`
- `request_payload`
- `expires_at`
- `used_at`

说明：

- 当前用于全服/批量发邮件与发物品确认流

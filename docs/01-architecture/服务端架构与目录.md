# 服务端架构与目录

## 1. 当前结构结论

当前服务端 `app` 目录已经完成一轮结构重构，真实主线收敛为：

```text
app/
  admin_support/
  bootstrap/
  core/
  game/
  ops/
  test_support/
  main.py
```

历史目录：

- `app/api`
- `app/modules`
- `app/schemas`
- `app/core` 顶层旧入口文件

都已经退出运行主线，其中大部分已直接删除。

当前对外接口路径包括：

- `/api/auth/*`
- `/api/game/*`
- `/api/test/*`
- `/api/admin/*`
- `/ops/api/*`

这轮重构主体仍然属于“内部结构重构”，`/ops/api/*` 是在此基础上新增的正式运维接口。

## 2. 各目录职责

### 2.1 `app/bootstrap`

全局启动装配层。

当前主要文件：

- `ApiRouter.py`

职责：

- 聚合 `game`、`test_support`、`admin_support` 的路由
- 聚合正式 `ops` 运维接口
- 给 `app/main.py` 提供统一的 `api_router`

它不承载具体业务逻辑，只负责顶层装配。

### 2.2 `app/core`

共享基础设施层。

当前按子域拆分为：

- `config/`
- `db/`
- `logging/`
- `resources/`
- `security/`
- `locks/`

职责边界：

- 配置读取
- 数据库模型与初始化
- 日志
- 仓库级资源文件
- 鉴权/密码/JWT/敏感词/校验
- 写锁与并发保护

不再放游戏语义很强的运行时编排代码。

### 2.3 `app/core/db`

数据库层。

职责：

- 数据库初始化与关闭
- Tortoise 模型定义

当前主要文件：

- `Database.py`
- `Models.py`

### 2.4 `app/game`

游戏主线业务域。

当前按层拆为：

- `api/`
- `application/`
- `domain/`
- `content/`
- `schemas/`

#### `app/game/api`

HTTP 路由与请求编排层。

示例：

- `AuthApi.py`
- `CultivationApi.py`
- `InventoryApi.py`
- `AlchemyApi.py`

职责：

- 接收请求
- 组装依赖
- 调用应用层/领域层
- 输出统一响应

#### `app/game/application`

游戏应用层。

示例：

- `GameContext.py`
- `Dependencies.py`
- `DailyReset.py`
- `InitPlayerInfo.py`
- `AntiCheatSystem.py`

职责：

- 运行时对象装配
- 公共写上下文
- 每日重置判定
- 初始化玩家数据
- 运行态与反作弊编排

它承接的是“游戏应用逻辑”，不是底层基础设施。

#### `app/game/domain`

领域层。

按模块划分：

- `account/`
- `player/`
- `cultivation/`
- `spell/`
- `inventory/`
- `alchemy/`
- `lianli/`
- `herb/`
- `task/`
- `mail/`

职责：

- 具体系统实现
- 数据读取器
- 算法与规则

#### `app/game/content`

游戏静态内容资源。

当前包括：

- `cultivation/realms.json`
- `spell/spells.json`
- `inventory/items.json`
- `alchemy/recipes.json`
- `lianli/areas.json`
- `lianli/enemies.json`
- `herb/herb_points.json`
- `task/tasks.json`

这是当前游戏数值与静态配置的真值目录。

#### `app/game/schemas`

按领域拆分的请求/响应结构定义。

示例：

- `AuthSchema.py`
- `CultivationSchema.py`
- `InventorySchema.py`
- `MailSchema.py`

### 2.5 `app/test_support`

测试支持域。

当前分为：

- `api/`
- `bootstrap/`

职责：

- 提供 `/api/test/*` 测试接口
- 提供测试账号初始化辅助

典型文件：

- `api/TestApi.py`
- `bootstrap/TestAccountSeeder.py`

### 2.6 `app/ops`

正式运维系统域。

当前按职责拆分为：

- `api/`
- `services/`
- `auth/`
- `audit/`
- `models/`

职责：

- 提供 `/ops/api/*` 正式运维接口
- 承接运维账号、运维 token、权限校验
- 承接审计日志和高危确认流
- 承接玩家查询、玩家管控、发放、登录闸门与白名单

### 2.7 `app/admin_support`

历史管理接口支持域。

当前主要文件：

- `api/AdminApi.py`

职责：

- 承接当前仍存在的 `/api/admin/*` 入口

说明：

- 这是对历史管理员接口的归位
- 它已不再挂在旧 `app/api` 下
- 当前仍保留 `/api/admin/*` 兼容入口
- 正式能力已经开始迁往 `app/ops`

### 2.8 `app/core/resources`

仓库级资源文件。

当前主要是敏感词词表：

- `sensitive_words_zh.txt`
- `sensitive_words_en.txt`

## 3. 当前启动链路

当前服务端启动链路如下：

```text
app/main.py
  -> app/bootstrap/ApiRouter.py
     -> app/game/api
     -> app/test_support/api
     -> app/admin_support/api
```

应用启动时还会执行：

- 敏感词系统初始化
- 数据库初始化
- 测试账号播种

## 4. 当前设计原则

这轮重构后，目录落点遵循以下原则：

1. 共享基础设施放 `core`
2. 玩家正式主线逻辑放 `game`
3. 测试专用能力放 `test_support`
4. 历史管理接口单独放 `admin_support`
5. 顶层聚合放 `bootstrap`
6. 不为了“目录好看”额外制造空层或伪概念

## 5. 后续边界

从目录重构角度，这轮已经完成。

后续如果继续推进，属于新的主题，而不是这轮未完成项：

1. 正式运维平台落地，逐步从 `admin_support` 过渡到 `ops`
2. 单独排查状态型波动测试
3. 如有必要，再对 `game/application` 或 `core/security` 做更细粒度拆分
